Ansible is the dominant agentless config-management tool in EU mid-market on-prem

## When Ansible fits

| Scenario | Ansible fit |
|---|---|
| On-prem Linux/Windows server config | excellent |
| Network device config (NX-OS, Juniper, ASA) | excellent |
| Cloud baseline config (AWS via boto3 modules, Azure, GCP) | very good |
| Kubernetes resources | use Kyverno instead (native) |
| Application policy (admission, runtime) | use OPA/Kyverno instead |
| Ad-hoc remediation across heterogeneous fleet | excellent |

Ansible is the right target for "I have 200 RHEL/Ubuntu/Windows servers in
on-prem datacenters and I need to configure them consistently per my policy."
Vanta and Drata cannot do this. Egide does, by emitting playbooks.

## Egide convention

Compiled Ansible playbooks bind back to the source Intent:

```yaml
---
# Generated by Egide compiler v0.1.0
# Intent: intent_db_backup_required v1.2.0
# Source pyramid: policy_data_protection_v3
# Normative anchor: iso27001-2022:A.8.13
- name: Enforce production database backup configuration
  hosts: production_databases
  become: true
  vars:
    egide_intent_id: intent_db_backup_required
    egide_intent_version: "1.2.0"
    egide_severity: error
    backup_frequency_hours: 24
    backup_retention_days: 30

  tasks:
    - name: Check current backup configuration
      community.postgresql.postgresql_query:
        query: "SELECT setting FROM pg_settings WHERE name = 'archive_mode'"
      register: archive_mode_check
      tags: [audit, egide]

    - name: Fail if archive_mode disabled (and severity == error)
      ansible.builtin.fail:
        msg: "Production database backup is not enabled â€” violates {{ egide_intent_id }}"
      when:
        - archive_mode_check.query_result.0.setting != 'on'
        - egide_severity == 'error'
      tags: [audit, egide]

    - name: Enable archive_mode
      community.postgresql.postgresql_set:
        name: archive_mode
        value: "on"
      when:
        - archive_mode_check.query_result.0.setting != 'on'
        - apply_remediation | default(false) | bool
      tags: [remediate, egide]
```

## Compiler responsibilities (services/compiler/generators/ansible)

For each TAI Intent compiled to Ansible:

1. Map `intent.selector.kinds` â†’ `hosts:` group(s) inferred from the tenant's
   Ansible inventory (J2). Example: `kind: database` â†’ host group `databases`.
2. Map `intent.selector.scope` â†’ group filter (`production_databases`).
3. Map `intent.required_state[]` â†’ a `Check` task + a `Fail` task (per-rule).
4. Map `intent.actions_on_violation`:
   - `audit` â†’ only `Check` + register; no `Fail` or modification.
   - `block_deployment` â†’ not applicable (Ansible is post-deploy); compiler
     emits a warning that `block_deployment` requires Rego/Kyverno target.
   - `auto_remediate` â†’ emit a remediation task tagged `remediate` (run only
     when `apply_remediation=true` in extra-vars, requiring explicit consent).
   - `notify_owner` â†’ emit a `mail` or `slack` task as a handler.
5. Always tag tasks: `audit` (read-only), `remediate` (modifies state),
   `egide` (universal). The user runs `--tags audit` for dry, `--tags remediate`
   for apply.
6. Emit a Molecule scenario for testing (when test fixtures are present).

## Two execution modes

Egide separates two modes:

### Audit mode (default)

```bash
ansible-playbook playbook.yml --check --diff --tags audit
```

Produces a JSON report (Egide custom callback plugin) with per-host findings.
This is what the continuous compliance journey (J4) ingests.

### Apply mode (J9 â€” gated)

```bash
ansible-playbook playbook.yml --tags egide,remediate -e apply_remediation=true
```

Apply mode requires:
- An approval workflow record in Egide (Pro/Enterprise).
- A documented exception window.
- Logs of every change persisted in `evidence_blobs`.

Egide ships a custom callback plugin that streams events to the API gateway
in real time during execution.

## Inventory contract

Egide expects the customer's inventory to be readable through one of:

- Static YAML inventory at `inventory/hosts.yml` (pulled via Git or upload)
- Dynamic inventory script returning JSON per Ansible spec
- Tower/AWX inventory API (Pro/Enterprise)

The compiler does NOT generate inventories â€” only playbooks. The user owns
their fleet definition.

## Testing

Each compiled playbook ships with:

- A Molecule scenario (`molecule/default/`) using Docker drivers for fast iteration.
- Fixtures: positive (compliant host) and negative (violating host).
- A Make target `make test` that runs `molecule test`.

The compiler's `Test()` method runs `molecule test` if Docker is available;
otherwise falls back to `ansible-playbook --check --syntax-check`.

## Don'ts

- Don't emit playbooks with `become: true` for tasks that don't need it.
- Don't hard-code credentials; use `ansible-vault` or external secret stores.
- Don't generate nested loops over inventory > 1000 hosts without forks tuning.
- Don't mix audit and remediate tasks without `tags`.
- Don't emit `shell:` or `command:` modules where a typed module exists
  (e.g., `community.postgresql.postgresql_set` over `shell: psql -c ...`).

## Sovereign hosting consideration

For customers with no internet access (air-gapped Enterprise), Egide bundles:

- A pinned Ansible 2.16+ in the install image
- Pre-downloaded Galaxy collections (community.postgresql, community.general,
  ansible.posix, ansible.windows, â€¦)
- An offline `ansible-galaxy` index

This avoids the trap of generating playbooks that require pulling collections
from the internet at execution.
