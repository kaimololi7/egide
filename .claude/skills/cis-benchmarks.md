# CIS Benchmarks — Operational guidance

CIS Benchmarks (Center for Internet Security) are configuration baselines for
operating systems, cloud platforms, and applications. They are widely used in
audits because they offer **prescriptive, measurable** controls (vs ISO 27001
which is principle-based).

Egide's compiler emits CIS-style audit checks as a Professional-tier target.
This bridges the abstract pyramid (ISO controls) with concrete machine-level
verification.

## Coverage

CIS Benchmarks cover:

| Category | Examples |
|---|---|
| Operating systems | Ubuntu 22.04 LTS, RHEL 9, Windows Server 2022, macOS 14 |
| Cloud | AWS, Azure, GCP, Oracle Cloud, Alibaba |
| Containers | Docker, Kubernetes, OpenShift |
| Applications | nginx, Apache, MySQL, PostgreSQL |
| Networks | Cisco, Juniper |
| Mobile | iOS, Android |

For Egide MVP, we focus on **5 high-value benchmarks**:
- CIS Ubuntu 22.04 LTS
- CIS RHEL 9
- CIS Windows Server 2022
- CIS Kubernetes 1.28+
- CIS Docker

## Levels

CIS distinguishes:

- **Level 1**: practical, minimal-impact controls (default).
- **Level 2**: defense-in-depth, may impact functionality (selected).
- **STIG-aligned profiles**: for US DoD environments (out of scope).

Egide compiler defaults to Level 1; users opt into Level 2 per intent.

## Output format options

The compiler emits CIS-style checks in three forms:

### 1. OpenSCAP / SCAP-compatible XML

For environments that already use SCAP (`oscap`):

```xml
<Rule id="xccdf_egide_rule_5_6_1_postgres_archive_mode" severity="medium">
  <title>Ensure PostgreSQL archive_mode is enabled</title>
  <description>Production databases must have archive mode enabled per
    intent_db_backup_required (ISO 27001 A.8.13).</description>
  <reference>iso27001-2022:A.8.13</reference>
  <check system="http://oval.mitre.org/XMLSchema/oval-definitions-5">
    <check-content-ref href="ovaldef.xml" name="oval:egide:def:1"/>
  </check>
</Rule>
```

### 2. Bash audit script (default — most portable)

```bash
#!/usr/bin/env bash
# CIS-style check generated by Egide
# Intent: intent_db_backup_required v1.2.0
# Source: policy_data_protection_v3 / iso27001-2022:A.8.13

set -euo pipefail
RULE_ID="egide_rule_5_6_1"
RESULT_FILE="${EGIDE_RESULT_FILE:-/var/log/egide/cis-results.json}"

archive_mode=$(sudo -u postgres psql -tAc "SHOW archive_mode" 2>/dev/null || echo "ERROR")

if [[ "$archive_mode" == "on" ]]; then
  result="pass"
else
  result="fail"
fi

cat >> "$RESULT_FILE" <<EOF
{"rule_id":"$RULE_ID","result":"$result","timestamp":"$(date -Iseconds)","detail":"archive_mode=$archive_mode"}
EOF
```

### 3. Inspec/Chef audit DSL (when customer already uses Chef)

```ruby
control 'egide_rule_5_6_1' do
  impact 0.7
  title 'Ensure PostgreSQL archive_mode is enabled'
  desc 'Production databases must have archive mode enabled (ISO 27001 A.8.13)'
  ref 'iso27001-2022:A.8.13'

  describe postgres_session('postgres', '', 'localhost').query("SHOW archive_mode") do
    its('output') { should eq 'on' }
  end
end
```

## Compiler responsibilities (services/compiler/generators/cis)

For each TAI Intent compiled to CIS:

1. Map `intent.selector.kinds` → target OS/platform inferred from inventory.
2. Lookup the **CIS Benchmark section** that corresponds (e.g., "PostgreSQL"
   = section 5.6, "User accounts" = section 1.4).
3. Generate audit script + a **failure remediation hint** (the CIS guide
   text, paraphrased).
4. Output a `manifest.json` that the customer can consume to drive automated
   audit cycles.

## Evidence integration

When the audit script runs (manually or via cron/scheduler), it writes
findings to a JSON log. The Egide pipeline service ingests this log:

- Per-rule pass/fail counts → KPI actuals.
- Per-host history → drift detection.
- Failed rules with severity error → continuous compliance alerts.

This ties CIS-level findings back to the pyramid's ISO/NIS2 controls — a
single failed CIS check propagates upward to "ISO 27001 A.8.13 not satisfied"
status.

## Distribution

## Don'ts

- Don't reproduce verbatim CIS guide text. CIS Benchmarks are
  **copyrighted by CIS**; we may reference rule IDs and structure but must
  paraphrase descriptions.
- Don't generate scripts that require root for read-only checks unless
  necessary; prefer least privilege.
- Don't set scripts to auto-remediate by default. Audit first, remediate
  separately (analogous to Ansible).
